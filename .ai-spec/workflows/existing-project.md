# 老项目接入工作流

> **注意**：AI-START.md §1.5 自动接入模式已处理大部分本文件描述的步骤。以下仅在非自动模式或用户明确要求时适用。

适用于已经运行、发布或积累了稳定约定的项目。

## 原则

不覆盖、不大爆炸重构、不把新规范追溯性强加到所有历史代码。先记录真实现状，再逐模块提升。

## 流程

1. 只读体检：代码结构、构建、测试、CI、配置、Git、AI 规则和文档。
2. 生成现状基线：哪些规则已满足、部分满足、不满足、暂不适用。
3. 找出业务规则来源和关键模块 Owner，不让 AI 自行发明规则。
4. 形成接入差异和风险排序，先处理密钥、权限、数据安全等高风险项。
5. dry-run 展示将新增、合并、保持不动的文件。
6. 备份已有 AI 配置；采用语义合并，禁止简单覆盖或 allow 权限取并集。
7. 分阶段启用：安全与交付 → 契约与测试 → 技术栈 → Skills 与治理。
8. 新代码立即遵守；旧代码在被修改或专项治理时逐步迁移。

## 兼容策略

冲突规则先记录为迁移项。需要暂时保留旧行为时，创建有到期时间的例外，不把临时状态写成永久规范。

## 回滚

接入改动使用独立分支或明确文件备份。回滚只撤销规范接入文件，不删除业务工作区或未提交改动。
