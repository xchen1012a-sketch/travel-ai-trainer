# 数据库变更与迁移规范

> 数据是公司资产。任何 DDL / DML / 数据迁移必须可审计、可分阶段，并有经过验证的恢复策略。

---

## 一、变更分类

| 类型 | 风险 | 流程 |
|---|---|---|
| **DDL 加字段（可空）** | 低-中 | 迁移脚本 + 锁/重写评估 + 恢复策略；风险取决于数据库版本和表规模 |
| **DDL 加字段（NOT NULL）** | 中 | 分两步：先加可空，回填，再改 NOT NULL |
| **DDL 删字段 / 改字段类型 / 改字段语义** | **高** | 多步灰度（详见下文） |
| **DDL 加索引** | 低-中 | 使用当前数据库版本支持的在线/并发建索引能力并验证锁影响 |
| **DDL 加表** | 低 | 标准迁移 |
| **DML 写入 / 回填** | 中 | 必须幂等 + 分批 + 进度记录 |
| **DML 删除 / 清理** | **高** | 必须有可验证恢复点 + 双人确认 + 范围/批次保护 |

---

## 二、迁移脚本规范

### 2.1 命名
- 按时间 + 序号：`V20260619_001__create_order_table.sql`
- 版本工具（Flyway / Liquibase / Alembic）按文件名顺序执行
- **不允许**改已上线的迁移文件（改了等于破坏版本一致性）

### 2.2 落点
- 放在模块资源目录：`src/main/resources/db/migration/`（按框架约定）
- 或单独 `migrations/` 目录

### 2.3 每个脚本必须有
- **注释头部**：变更目的、作者、日期、关联 issue/ticket
- **正向 SQL**：实际变更
- **恢复方案**：安全可逆时提供回滚脚本；否则提供兼容切换、备份恢复或前向修复步骤

```sql
-- V20260619_001__add_index_on_order_user_id.sql
-- 目的：加速 user_id 维度查询
-- 关联：TICKET-1234
-- 作者：alice  日期：2026-06-19

-- 示例（数据库方言不同，具体语法按项目实际）
CREATE INDEX CONCURRENTLY idx_orders_user_id ON orders(user_id);

-- 回滚（单独执行）
DROP INDEX CONCURRENTLY idx_orders_user_id;
```

## 三、破坏性变更（删字段 / 改类型）的灰度步骤

**严禁一步删字段**。必须分多步：

```
Step 1（v1）：DB 加新字段（可空）
   ↓ 部署 → 验证
Step 2（v2）：双写（同时写旧字段和新字段）
   ↓ 部署 → 验证 → 观察 N 天
Step 3（v3）：读切到新字段，停止写旧字段（旧字段保留）
   ↓ 部署 → 验证 → 观察 N 天
Step 4（v4）：清理旧字段（DDL 删除）
```

**改字段类型**同理（新建 → 双写 → 切读 → 删旧）。

**改字段语义**（同名不同含义）也算破坏性变更，需走灰度。

## 四、数据回填规范

### 4.1 幂等
- 使用当前数据库方言支持的原子 UPSERT、唯一约束或幂等进度键
- 重跑不产生重复数据；并发执行也不能突破唯一性约束

### 4.2 分批
- 单次操作 ≤ 1000 行
- 每批之间 sleep（避免压垮 DB）
- 记录进度到 `migration_progress` 表（断点续跑）

```sql
-- 反例：一次性更新百万行（会对大表加锁，阻塞生产）
UPDATE orders SET status = 'archived' WHERE created_at < '2025-01-01';

-- 正例：分批更新（具体语法按数据库方言实现）
UPDATE orders SET status = 'archived'
WHERE id IN (
  SELECT id FROM (
    SELECT id FROM orders
    WHERE created_at < '2025-01-01' AND status != 'archived'
    LIMIT 1000
  ) t
);
-- 重复执行直到 affected_rows = 0
```

### 4.3 进度可观测
- 每批打日志（已处理 N / 总 M / 速度）
- 失败可重试，不污染已成功部分

## 五、删除 / 清理规范（高风险）

### 5.1 删除前
- **恢复点**：使用数据库备份/快照、可校验导出、影子表或其它能保留所需结构与数据的机制
- **双人确认**：写 SQL 的人 + 审核 SQL 的人
- **范围校验**：先 `SELECT COUNT(*)` 看影响行数，避免误删
- **范围保护**：必须有明确条件、影响行数预览、批次上限和停止条件；具体 SQL 按数据库方言实现

### 5.2 删除时
- 分批删（同回填规范）
- 业务低峰执行
- 监控 DB 负载，超阈值暂停

### 5.3 删除后
- 验证业务功能正常
- 备份表保留 N 天后清理
- 写事后审计日志

## 六、生产 DDL 执行流程

```
1. 在开发环境跑通迁移 + 恢复/兼容路径
2. Code Review（重点看锁影响、数据完整性和恢复方案是否真实可用）
3. staging 环境跑一次完整迁移
4. 生产：先备份（全量 / 增量）
5. 生产：执行迁移
6. 生产：验证（应用启动 / 关键接口 / 数据抽样）
7. 如异常：按预案停止、切换、回滚或前向修复，并通知相关负责人
```

**禁止**：
- 跳过 staging 直接上生产
- 生产执行时无人监控
- 没有经过评审的恢复策略就执行 DDL

## 七、表设计基础（避免后续迁移）

新表上线前自检，减少未来 DDL：

- [ ] 字段类型合理（不用 VARCHAR 装日期、不用 FLOAT 装钱）
- [ ] NOT NULL + 默认值（除非真的需要 NULL 语义）
- [ ] 主键明确（自增 ID / UUID，不缺主键）
- [ ] 关键查询字段加索引
- [ ] 外键关系清楚（即使不用 FK 约束，也要文档化）
- [ ] 时间字段齐全（`created_at` `updated_at`）
- [ ] 软删字段（`deleted_at` 或 `deleted`）按需加
- [ ] 多租户字段（`tenant_id`）按需加
- [ ] 字符集 / 排序规则与项目数据库规范一致，并验证大小写、重音和多语言语义

## 八、变更检查清单（MR 前）

- [ ] 迁移脚本 + 恢复策略已写；安全可逆时包含回滚脚本
- [ ] 脚本有注释头部
- [ ] 在开发环境跑通正向迁移 + 适用的恢复/兼容路径
- [ ] 在 staging 跑通正向
- [ ] 大表变更评估了锁、复制延迟、磁盘和在线变更能力
- [ ] 破坏性变更走灰度（多步）
- [ ] 数据回填幂等 + 分批
- [ ] 删除操作有可验证恢复点 + 范围/批次保护
- [ ] 影响评估（停机 / 锁表 / 性能）
