# 权限化与配置化规范

> 适用于任何有"谁能做什么 / 谁能看什么"需求的项目。
>
> 核心原则：**权限在后端做，前端只做体验；不硬编码身份 ID；规则可配置可审计**。

---

## 一、三大权限模型

| 模型 | 全称 | 适用场景 |
|---|---|---|
| **RBAC** | Role-Based Access Control | 角色固定、按角色分组授权（最常见） |
| **ABAC** | Attribute-Based Access Control | 基于属性（时间/IP/数据敏感度）动态授权 |
| **ACL** | Access Control List | 细粒度到单资源的白名单（文件系统/对象存储） |

**推荐**：默认 RBAC，特殊场景叠加 ABAC（如"工作日 9-18 点才能访问"）。

---

## 二、功能权限（菜单 + 按钮）

### 2.1 菜单权限
- 受保护的功能入口必须有明确授权策略；公开页面不强制菜单权限，但必须显式标记为公开
- 菜单通过角色 → 权限关联控制可见性
- 用户登录后只看到自己有权限的菜单

### 2.2 按钮权限
- 产生受保护写入、导出或审批的动作必须映射到后端授权能力；纯本地或公开动作不机械增加权限码
- 权限码命名规范：`{module}:{resource}:{action}`
  - 例：`sales:score:save` / `system:user:create` / `finance:report:export`
- **禁止**只靠前端隐藏按钮代替权限（用户可绕过前端直接调 API）

### 2.3 后端强制校验
- 受保护 API 必须在**入口处**校验权限（注解 / 中间件 / 拦截器）
- 登录、注册、Webhook 等公开或外部入口必须进入显式允许清单，并配置专用认证、签名、限流或滥用防护
- 前端隐藏按钮 ≠ 权限，只是体验优化
- 权限缺失时返回标准 403，不泄露"为什么没权限"

## 三、数据权限（行级）

### 3.1 数据范围
涉及客户、订单、KPI、日报、财务、统计时，必须考虑：
- **公司/租户**：默认隔离，跨租户访问需明确授权
- **部门**：本部门 / 本部门+下属部门 / 指定部门
- **角色**：管理员 / 主管 / 普通员工 / 访客
- **用户**：本人 / 指定用户组
- **时间**：当前 / 历史 / 自定义范围

### 3.2 数据范围解析
- 主管查看下属数据**必须走系统组织能力**（部门树 / 上下级关系），不能硬编码下属 ID
- 数据范围在**后端 SQL/ORM 层**强制注入（拦截器 / Repository 基类）
- 超管豁免、租户隔离、角色数据范围**不得被绕过**

### 3.3 越权检查（IDOR 防护）
- 用户传入的 `resource_id` 必须校验**归属**
- 例：用户 A 调 `GET /api/orders/123`，必须校验 123 是否属于 A（或 A 有权限的范围）
- 不校验 = 水平越权漏洞

## 四、权限矩阵模板

新功能上线前必须填这张表，避免遗漏：

| 资源 \ 角色 | 超管 | 管理员 | 主管 | 员工 | 访客 |
|---|---|---|---|---|---|
| 订单.查看 | ✅ 全部 | ✅ 本部门树 | ✅ 本部门+下属 | ✅ 本人 | ❌ |
| 订单.创建 | ✅ | ✅ | ✅ | ✅ | ❌ |
| 订单.删除 | ✅ | ✅ | ❌ | ❌ | ❌ |
| 订单.导出 | ✅ | ✅ | ✅ | ❌ | ❌ |

填完后转化为：
- **菜单权限**：管理员 / 主管 / 员工 → 菜单可见
- **按钮权限码**：`order:view` `order:create` `order:delete` `order:export`
- **数据范围**：每角色的 SQL where 条件

## 五、配置化（不硬编码）

### 5.1 禁止硬编码的内容
- 用户 ID / 员工 ID
- 部门 ID / 角色 ID
- 公司 ID / 租户 ID
- 具体人名（"张三审批"）
- 业务阈值（如"KPI > 100 算优秀"中的 100）
- 第三方域名 / 端口 / 路径

### 5.2 应该用配置/数据化
- 身份/组织：从系统组织表读取（部门树 / 角色表 / 用户表）
- 业务阈值：配置表 / 配置项 / 规则引擎
- 第三方接入：环境变量 / 配置中心

### 5.3 可扩展性
- 只有项目画像包含多组织或多租户需求时，才要求扩展到多公司、多部门和多角色
- 单租户项目不提前引入多租户复杂度，但避免把可变组织标识散落硬编码

## 六、审计与日志

### 6.1 操作日志（必有）
- 写操作（创建 / 修改 / 删除 / 审批 / 导出）必须有日志
- 日志含：**谁、何时、从哪（IP）、对什么、做了什么、改前值、改后值、结果**

### 6.2 敏感字段追踪
- 修改手机号、邮箱、权限、角色、金额等敏感字段必须可追溯
- 推荐用审计表 + 字段级 diff 记录

### 6.3 外部写入审计
- 外部系统 / 机器人 / 批量导入的写入必须记录：
  - 来源标识
  - 幂等键
  - 处理状态（成功 / 失败 / 部分成功）
  - 失败原因
  - 接收时间 / 处理时间

> 审计日志详尽配置见 `observability.md` §二。

## 七、最小权限原则

- 默认**不给**权限，需要时**显式授予**
- 角色权限定期 review，移除不再需要的权限
- 临时权限（debug / 临时接管）必须有过期时间
- 高危操作（删数据 / 改权限 / 改配置）要求**双人确认**或**审批流**

---

## 八、权限变更检查清单

新增或修改权限时：

- [ ] 权限矩阵已更新（哪些角色有什么）
- [ ] 菜单 + 权限码已配置
- [ ] 后端 API 强制校验（不依赖前端）
- [ ] 数据范围已注入（SQL/ORM 层）
- [ ] 越权检查已加（IDOR 防护）
- [ ] 操作日志已记录
- [ ] 回滚方案已准备（如何撤销新权限）
