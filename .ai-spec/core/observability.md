# 可观测性规范（Observability）

> 看不见的系统调不了。日志 / Metrics / Trace 三件套缺一不可。

---

## 一、三大支柱

| 支柱 | 回答什么 | 工具示例 |
|---|---|---|
| **日志（Logging）** | 发生了什么？为什么错？ | ELK / Loki / CloudWatch |
| **指标（Metrics）** | 现在状态如何？趋势怎样？ | Prometheus / Grafana / Datadog |
| **追踪（Tracing）** | 一次请求经过哪些服务？哪里慢？ | Jaeger / Zipkin / OpenTelemetry |

加一个：
| **健康检查（Health）** | 服务还活着吗？依赖正常吗？ | 项目定义的 liveness/readiness 信号 |

---

## 二、日志规范

### 2.1 日志级别

| 级别 | 用途 | 默认是否收集 |
|---|---|---|
| **ERROR** | 异常 / 失败 / 需要立即关注 | ✅ |
| **WARN** | 可疑但能恢复（重试成功 / 降级 / 接近阈值） | ✅ |
| **INFO** | 业务关键事件（登录 / 下单 / 支付） | ✅ |
| **DEBUG** | 排查问题时用的细节 | ❌（生产默认关） |
| **TRACE** | 极细粒度（每行 SQL / 每次循环） | ❌（永不进生产） |

### 2.2 日志结构化
- **必须**用 JSON 结构化日志（不是字符串拼接）
- 字段标准化：`timestamp` `level` `service` `trace_id` `user_id` `event` `message` `...custom`

```json
{
  "timestamp": "2026-06-19T10:30:00Z",
  "level": "INFO",
  "service": "order-service",
  "trace_id": "abc123",
  "user_id": 10104,
  "event": "order.created",
  "order_id": 5678,
  "amount": 99.00,
  "message": "Order created"
}
```

### 2.3 日志禁忌
- **不**记录密钥 / 密码 / Token / 完整身份证号 / 完整银行卡号
- **不**记录用户密码（即使是哈希）
- **不**用 `print()` / `console.log()` 替代日志框架
- **不**在循环里打日志（用批量 / 聚合）
- **不**打"调试日志"留在代码里（DEBUG 级别也要写明白意图）

### 2.4 必打日志的事件
- 认证成功 / 失败
- 权限拒绝
- 写操作（创建 / 修改 / 删除）
- 外部 API 调用（入参 / 出参 / 耗时 / 状态码）
- 异常 / 错误（含 stack trace）
- 关键状态流转（订单：pending → paid → shipped）

## 三、指标规范

### 3.1 必有指标

| 类型 | 指标 |
|---|---|
| **流量** | QPS / 请求量 / 活跃用户数 |
| **延迟** | P50 / P95 / P99 响应时间 |
| **错误** | 错误率 / 5xx 数量 / 异常数 |
| **饱和度** | CPU / 内存 / 磁盘 / 连接池 / 队列长度 |
| **业务** | 订单量 / 支付成功率 / 转化率 |

> USE 方法（Utilization / Saturation / Errors）+ RED 方法（Rate / Errors / Duration）是基础。

### 3.2 命名约定
- `{service}_{metric}_{unit}`
- 例：`order_api_latency_ms_p99` / `payment_success_rate`

### 3.3 不应有
- 不记录每个用户每次请求（用聚合，不用明细）
- 不把 PII（个人身份信息）当 label

---

## 四、追踪规范

- 入口生成 `trace_id`，跨服务透传（HTTP header / MQ metadata）
- 关键调用链有 span：DB 查询 / 外部 API / 缓存读写
- span 含：开始时间 / 耗时 / 状态 / 关键参数（脱敏）
- 慢查询 / 慢调用自动标记（> 500ms 标黄，> 2s 标红）

---

## 五、健康检查

```
GET /health       → 进程活着（返回 200）
GET {readiness-path} → 必要依赖达到可接流量状态
```

- 部署系统使用项目定义的 readiness 信号判断是否接流量
- 健康检查不做复杂业务逻辑（< 50ms 返回）
- 依赖检查用超时保护（连不上立即返回 503，不等）

## 六、告警规范

### 6.1 告警分级
| 级别 | 触发条件 | 响应 |
|---|---|---|
| **P0** | 服务不可用 / 数据丢失 | 立即电话 |
| **P1** | 核心功能受损 / 错误率 > 5% | 立即处理 |
| **P2** | 部分功能异常 / 性能下降 | 工作时间内处理 |
| **P3** | 提醒类（接近阈值） | 看到再说 |

### 6.2 告警规则
- **可执行**：告警必须告诉值班人"看哪里、做什么"
- **不噪**：一个告警 24h 内不超过 3 次，超过即合并或重设阈值
- **可恢复**：异常消失后自动 resolved
- **不告警**：日志里查得到的、不影响用户的、调试用的
