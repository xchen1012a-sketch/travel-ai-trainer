# 架构原则与模块边界

> 通用方法论，与语言/框架无关。具体技术栈补丁见 `../stacks/`。

## 二、模块边界原则

### 2.1 单一职责
- 一个模块 / 包 / 类只做一件事
- 模块名应该能直接回答"它做什么"，不需要解释

### 2.2 显式依赖
- 模块之间的依赖必须显式（import / dependency injection / 接口契约）
- **禁止隐式依赖**：全局变量、反射查找、字符串 key 调用

### 2.3 依赖方向（强约束）

依赖必须沿一个方向流动，**严禁循环依赖**：

```
入口层（Controller / CLI / Route Handler）
   ↓
应用层 / Service（业务编排）
   ↓
领域层 / Domain（业务规则、状态流转）
   ↓
基础设施层（DB / 外部 API / 文件 / 消息队列）
```

- 上层可以依赖下层
- 下层**不能**反向依赖上层
- 同层之间尽量不直接依赖，通过接口或事件解耦

### 2.4 端口适配器模式（推荐）

外部系统（DB、第三方 API、消息队列）一律抽象为**端口（Interface）**，具体实现是**适配器（Adapter）**：
- 业务逻辑依赖端口，不依赖具体实现
- 切换 MySQL → PostgreSQL 只改适配器，不动业务
- 测试时端口可被 mock

## 三、标准分层模板

新增业务按以下结构（具体包名按项目框架和现有代码习惯调整）：

```text
entry/                 # Controller / CLI / Route Handler
  ├── {feature}/
  └── {feature}/dto/   # 入参出参 VO（如有独立层）

service/               # 业务编排、事务、状态流转
  ├── {feature}/
  ├── {feature}/calculator/   # 纯计算
  └── {feature}/helper/       # 辅助

domain/                # 领域模型、领域规则（可选）
  └── {feature}/

infrastructure/        # 端口适配器（DB Repository / 外部 API Client / MQ）
  ├── persistence/{feature}/   # DO + Mapper / Entity + Repository
  ├── external/{feature}/      # 第三方 API client
  └── messaging/{feature}/     # 消息生产/消费

enums/                 # 枚举
config/                # 配置类
```

> 简单项目可以合并 service + domain；复杂项目必须分开。

## 四、新增代码落点决策树

```
要新增一个功能
    │
    ├─ 是系统/通用能力（用户/角色/权限/租户/部门/文件/日志）？
    │     → 复用现有系统模块，不新建
    │
    ├─ 是业务功能？
    │     → 落到对应业务模块的 {feature}/ 子包
    │     → 跨模块复用的 DTO/事件/通用服务 → biz-common / shared
    │
    ├─ 是新业务域（现有模块装不下）？
    │     → 先说明"为什么不能落在现有模块"
    │     → 再按本文件三层结构新建模块
    │
    └─ 是框架/基础设施变更？
          → 默认不动；动了必须说明理由 + 用户确认
```

## 五、禁止事项

- **不**自创新的分层方式（必须按上面三层或框架习惯）
- **不**让 Controller / Route Handler 直接写复杂业务逻辑
- **不**让 Controller / Route Handler 直接操作 DB / 外部 API
- **不**把外部接入逻辑（webhook、机器人、第三方回调）和核心业务逻辑混写
- **不**为一个页面需求修改框架层或共享基础层
- **不**产生循环依赖（包间、模块间）
- **不**跨模块直接读对方表/状态（必须通过对方 Service 或 API）

---

## 六、依赖检查（每次新增/修改依赖时自检）

- [ ] 这个依赖方向是否符合"上层依赖下层"？
- [ ] 是否引入了循环依赖？（可用 `madge` / `dependency-cruiser` / IDE 工具检查）
- [ ] 是否绕过端口直接依赖具体实现？
- [ ] 是否跨模块直接读对方内部状态？
- [ ] 这个新模块/包是否真的需要存在（能不能落到现有模块）？
