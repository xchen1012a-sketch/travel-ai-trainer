# 验证与交付规范

> 每次代码修改后必须按本文件交付。**未经验证的"完成"等于没完成**。

---

## 一、Definition of Done

任务真正完成的必要条件（不适用项必须能说明原因）：

- [ ] 本地编译/构建通过
- [ ] 改到的测试已运行通过
- [ ] 新增功能有对应测试
- [ ] 手动验证正常路径 + 至少一个边界
- [ ] 已给出 **改动 / 验证 / 风险 / 回滚** 四件套
- [ ] 涉及契约的，已同步 `docs/contracts/`
- [ ] 涉及 DDL 的，已提供回滚 SQL 或恢复策略

---

## 二、验证要求

| 类型 | 何时跑 |
|---|---|
| 编译/构建 | 适用时必跑 |
| 静态检查 | 改到 .ts / .java / .py 时 |
| 单元测试 | 改到的模块必跑 |
| 集成测试 | 改到 API / DB 层时 |
| 接口自测 | 改到接口时必跑 |

**禁止用以下话术替代验证**：

- "应该可以"
- "看起来没问题"
- "理论上没问题"
- "等下我验证一下"（然后没验证）

---

## 三、交付报告

### 3.1 默认轻量版（大部分任务用这个）

```markdown
## 交付说明
- **目标**：一句话说清本次解决什么
- **改动**：文件清单 + 每个文件改了什么
- **验证**：命令 + 真实结果（✅/❌）
- **风险/未验证**：可能踩的坑 + 没验证的项 + 原因
- **回滚**：怎么回到改前状态
- **下一步**（可选）：后续动作
```

### 3.2 完整版（仅跨模块 / 高风险 / 正式发布用）

在轻量版基础上补充以下影响面分析（无影响写"无"）：

- 后端影响（接口/服务/数据/性能）
- 前端影响（页面/组件/路由/状态）
- 数据库影响（DDL/迁移/索引）
- 权限影响（菜单/权限码/数据范围）
- 外部集成影响（第三方 API/Webhook/MQ）

---

## 四、回滚要求

任何改动必须有恢复策略。优先回滚；不可逆或回滚风险更高时，使用前向修复或备份恢复。

| 类型 | 要求 |
|---|---|
| 代码 | commit hash + `git revert` 或 `git checkout <hash> -- <files>` |
| 数据库 | 可逆时给逆向迁移；不可逆 DDL 必须事前说明恢复路径、兼容窗口和批准人 |
| 配置 | 配置项名 + 默认值 + 修改前值 |
| 权限 | 菜单/权限码/数据范围如何撤销，提供 SQL 或操作步骤 |

---

## 五、合并前置检查

- [ ] 适用 DoD 项已满足，或说明缺口和补齐方式
- [ ] 无未处理的阻断风险；CRITICAL 必须关闭
- [ ] 分支与目标分支无冲突
- [ ] CI 全绿
- [ ] 契约变更已同步调用方
- [ ] DDL 已在 staging 跑过
- [ ] 回滚方案写在 MR 描述里
