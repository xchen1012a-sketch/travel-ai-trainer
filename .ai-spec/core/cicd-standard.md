# CI/CD 规范

> 自动化是质量的底线。能机器做的不要靠人记忆。

---

## 一、分支模型

分支模型属于 `PROJECT` 决策，记录在 `ai-spec.yaml`，不得由模板强制所有项目使用 Git Flow。

| 模型 | 适用场景 | 必要保护 |
|---|---|---|
| Trunk-based | 高频集成、自动化成熟 | 主干保护、短分支、CI、渐进发布 |
| GitHub flow | 持续交付的服务或应用 | 默认分支保护、PR review、CI |
| Git Flow | 有明确版本列车或多版本维护 | 主分支和发布分支保护、回合并策略 |
| Monorepo 定制 | 多模块、按影响范围构建 | CODEOWNERS、路径门禁、受影响测试 |

无论采用哪种模型，受保护分支、紧急修复回流方式和发布来源都必须明确。

## 二、CI 流水线

```yaml
触发：项目配置的 PR/MR、主干或发布事件

Stages:
  1. checkout
  2. install          # 装依赖
  3. lint             # 静态检查 / 风格
  4. typecheck        # 类型检查（如有）
  5. build            # 编译 / 构建
  6. test:unit        # 单元测试
  7. test:integration # 集成测试
  8. security-scan    # 依赖 + 代码扫描
  9. coverage-check   # 覆盖率门槛
 10. (optional) e2e   # 关键流程 E2E
```

### 默认门槛（项目可按风险调整并记录）
- Lint 0 error（warning 可容忍）
- Type check 0 error
- Build 成功
- 本次要求执行的测试全部通过；隔离测试必须有 issue、Owner 和到期时间
- 覆盖率满足项目基线；老项目优先保证变更代码覆盖和整体不恶化
- 已确认可利用的 CRITICAL 漏洞阻塞发布；HIGH 必须修复或有批准的限时例外与补偿控制
- E2E（如属于当前门禁）通过；不稳定测试不得靠无限重试掩盖

### 失败处理
- 失败 job **阻塞合并**
- 失败日志必须可读（不要把 1000 行 stack 全 dump 到终端，挑关键）
- 失败可重试（网络抖动 / flaky test），但同一失败重试 ≤ 3 次

## 三、CD 流水线（部署）

### 3.1 环境分级

| 环境 | 部署方式 | 谁能触发 |
|---|---|---|
| dev | 项目配置的开发事件 | 开发者或自动化 |
| staging | 项目配置的集成/发布候选事件 | 自动或发布负责人 |
| **prod** | 受保护发布来源 + 项目批准策略 | 授权发布者或受控自动化 |

生产发布必须有明确的授权、审计、渐进发布和自动停止条件。高风险系统默认人工批准；成熟的低风险持续部署可由项目正式批准自动化。

### 3.2 部署策略

| 策略 | 适用 | 复杂度 |
|---|---|---|
| **Rolling** | 无状态服务、向后兼容 | 低 |
| **Blue/Green** | 需要快速回滚 | 中 |
| **Canary** | 高风险变更、想观察 | 高 |
| **Recreate** | 仅开发/测试环境 | 低 |

### 3.3 健康检查
- 部署后等待项目定义的 readiness/健康信号和关键冒烟检查通过才算成功
- 失败自动回滚到上一个版本
- 部署窗口：业务低峰

## 四、制品管理

- 发布制品必须可追溯到不可变版本和源码 commit；是否同时使用版本号与 commit tag 由制品系统决定
- 制品仓库保留时间由审计、回滚窗口、法规和成本共同决定，生产制品至少覆盖支持与恢复周期
- 对外发布、生产部署或高风险制品应生成并保存 SBOM（软件物料清单）

---

## 五、密钥管理

- CI/CD 密钥**必须**放 Secret Manager（GitHub Actions Secrets / GitLab CI Variables / Vault）
- **绝不**写在 YAML 明文
- **绝不**在日志输出（CI 工具会自动 mask，但要确保不传给第三方 step）
- 不同环境用不同密钥（dev 密钥不能复用到 prod）

---

## 六、回滚

### 6.1 应用回滚
- 重新部署上一个版本的制品 tag
- 数据库变更**不自动回滚**（DDL 回滚必须人工评估）
- 回滚或恢复时间目标由项目 RTO/SLO 定义并定期验证

### 6.2 回滚演练
- 按项目关键等级和 RTO 设定恢复演练频率，并在重大架构/部署变化后补充演练
- 验证回滚流程 + 制品仓库保留策略有效

## 七、流水线检查清单

新功能上线前：

- [ ] MR 触发 CI 跑通
- [ ] 所有必过门槛达标
- [ ] 部署到 staging 验证
- [ ] 制品已推送且打了双 tag
- [ ] 健康检查端点正常
- [ ] 回滚方案已演练 / 评估
- [ ] 生产部署窗口已确认
- [ ] 监控告警已就位
