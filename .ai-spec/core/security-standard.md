# 安全红线

> 任何任务都必须遵守。普通业务需求不能静默覆盖安全底线；确需偏离时必须走正式例外、风险接受和补偿控制流程。

---

## 一、密钥管理

- 真实第三方密钥（数据库密码、API Key、Token、OAuth Secret、私钥）**一律用环境变量或密钥管理器**占位（`${ENV_VAR}` / `process.env.X` / `@Value("${X}")`）
- **绝不**写进版本控制文件（源码、配置文件、Dockerfile、CI YAML、README、注释、示例代码）
- **绝不**在日志、错误消息、API 响应、聊天工具中输出密钥
- 启动时校验所需密钥是否存在，缺失则 fail fast
- 任何疑似泄露的密钥立即轮换，不存侥幸

### 凭证文件黑名单（禁止读取、禁止外泄）

- `*.pem` `*.key` `*.p12` `*.keystore`
- `.env.local` `.env.production` `.env.*.local`
- `*credential*` `*secret*`
- `~/.ssh/**` `~/.aws/**` `~/.config/gcloud/**` `~/.kube/**`
- `.npmrc`（含 token）`.pypirc` `.netrc`

## 二、数据安全

### 2.1 写操作必须幂等
- 使用数据库唯一约束、幂等键、事务或数据库方言支持的原子 UPSERT；单纯“先查后写”不能抵抗并发竞争
- 重复消息、重复导入、重复点击不能产生重复数据
- 涉及钱、库存、积分等的写操作必须**强幂等**（幂等键 + 状态机/唯一约束）

### 2.2 删除红线
- **不删现网数据**
- 删除优先用**逻辑删**（`deleted = 1` / `status = 'archived'`）
- 物理删除、清空表、删字段必须**用户明确确认**
- 清理任务必须先预览影响范围，并使用明确条件、批次上限、事务/检查点和数据库方言支持的安全分批方式

### 2.3 数据导出
- 敏感字段（手机号、身份证、邮箱、地址）导出前脱敏
- 大数据量导出走流式（EasyExcel / cursor），避免 OOM
- 导出文件加权限控制 + 水印（如可能）

---

## 三、输入验证（边界处必验）

所有从外部进入系统的数据**必须验证**：
- HTTP 请求参数（query / body / path / header）
- 文件上传（类型、大小、内容）
- 第三方回调 / Webhook
- 消息队列消费
- 命令行参数

**验证项**：类型 / 长度 / 范围 / 格式 / 业务合法性。**Fail fast**，错误信息不泄露系统细节。

---

## 四、注入防护（OWASP Top 10 摘要）

| 风险 | 防护 |
|---|---|
| SQL 注入 | 永远用参数化查询 / Prepared Statement；**禁止**字符串拼接 SQL |
| XSS | 输出转义；禁止 `innerHTML` / `dangerouslySetInnerHTML` 未净化使用；CSP 头 |
| CSRF | 写操作用 Token / SameSite Cookie |
| SSRF | 外部 URL 请求前校验协议 + 域名白名单；禁止直接请求用户传入的 URL |
| 路径遍历 | 文件路径校验、白名单字符、不信任用户输入拼路径 |
| XXE | XML 解析禁用外部实体 |
| 反序列化 | 不反序列化不可信数据；用 JSON 不用二进制序列化 |
| 命令注入 | 不拼 shell 命令；用 SDK / 参数数组 |

---

## 五、认证与授权

- 受保护的写操作 API 必须鉴权；注册、登录、Webhook 等公开写入口必须显式登记并使用限流、签名或专用防滥用控制
- 每个数据查询必须有**数据范围校验**（用户只能看自己有权限的数据）
- 越权检查（IDOR）：用户传入的 resource_id 必须校验归属
- 权限判断在**后端**做，前端只做体验层控制（隐藏按钮 ≠ 权限）
- 详细规则见 `permission-standard.md`

---

## 六、生产与运行

- **不主动推云端**
- 项目配置标记的生产环境未经明确“部署/上线”授权**不动**
- 改完代码只重启本地服务验证，不碰服务器
- 不在生产数据库直接执行写 SQL（如必须，先备份 + 双人确认 + 审计日志）

---

## 七、Git 安全

- **不**直推项目配置或仓库策略标记为受保护的分支
- **不** `git push --force`（除非自己刚开的 feature 分支、且确认无他人协作）
- **不**用破坏性命令覆盖用户未提交改动（`git reset --hard` 前先 stash / 确认）
- **不**执行 `rm -rf` 关键目录
- **Git 提交硬性门禁**：暂存区必须检查，不能直接 `git add -A` 后无差别提交；必须逐项确认进入 commit 的文件。
- **不**把临时文件 / IDE 配置 / `.env*` commit 进 git（`.gitignore` 必须配齐）
- **提交前过滤**：暂存区检查，排除配置文件（`.env` / IDE 配置 / 本地设置 / 构建产物），只提交纯代码（源码、测试、迁移、锁文件、文档）；CI 配置文件可作为特例提交，但不得含密钥
- **`git pull` 行为**：`git pull` = `git fetch` + `git merge`，默认将远程最新提交与本地最新提交合并；有冲突时 Git 标记冲突文件，需手动解决后提交 merge commit。可改用 `git pull --rebase` 将本地提交变基到远程最新之上，保持线性历史

---

## 八、依赖安全

- 定期跑依赖扫描（`npm audit` / `pip-audit` / `mvn dependency-check` / Dependabot / Snyk）
- 新增依赖前查 issue 数、最近 commit、是否有已知 CVE
- 锁文件（`package-lock.json` / `pnpm-lock.yaml` / `Cargo.lock` / `go.sum`）必须入 git
- 不安装来历不明 / 已弃用的包

---

## 九、高风险禁止事项（除非用户明确确认）

未经用户明确确认，**禁止**：

1. 直接操作生产数据库
2. 删除表、删除字段、清空数据
3. 绕过权限或临时关闭鉴权"调试"
4. 修改登录、认证、数据权限核心逻辑
5. 改变 KPI、财务、订单、客户归属等业务口径
6. 为了通过编译而注释掉关键逻辑（try/catch 吞异常、空实现等）
7. 把敏感信息（Token / 密码 / API Key / 数据库密码）写入代码
8. 修改外部接口鉴权但不说明风险
9. 修改定时任务 / 消费者但不说明触发对象和频率
10. 批量发送消息（短信 / 邮件 / 推送）但不做范围控制
11. 未验证就声称完成

---

## 十、安全事件响应

发现安全问题（疑似泄露 / 越权 / 漏洞）时：

1. **立即停止**当前任务
2. 使用项目可用的安全审查能力或人工安全负责人复核
3. **CRITICAL** 级问题必须先修复再继续
4. 轮换任何可能暴露的密钥
5. 审查整个代码库是否有同类问题
6. 通知相关负责人
