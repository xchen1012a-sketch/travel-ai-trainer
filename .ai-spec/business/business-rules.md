# 业务规则（AI 实时维护）

> 本文件是 AI 实时维护的**活动文档**，不需要用户手动填。详见 `AI-START.md` §1.6。
>
> **AI 维护规则**：
> - 老项目接入时，AI 扫描代码 / commit / 文档按章节填入
> - 新项目接入时，AI 根据用户口述按章节起草
> - 日常代码修改涉及业务逻辑时，AI 自动追加 / 升级 / 标冲突
>
> **每条规则必须带标记**：
> - `[来源: 代码/用户/推断]`
> - `[可靠度: 高/中/低]`
> - `[📌 用户确认]`（用户钉死的规则，AI 永不覆盖）
> - `[⚠️ 冲突 YYYY-MM-DD]`（代码与规则不一致，等用户裁决）
>
> **用户参与**：零日常维护。仅在出现 `[⚠️ 冲突]` 时裁决，或主动用 `[📌 用户确认]` 钉死关键规则。

## 三、组织身份规则（如有组织概念）

### 3.1 身份来源
- 身份（人是谁）以什么为单一来源？
  - [ ] 自建用户系统
  - [ ] 第三方身份（企业微信 / 钉钉 / 飞书 / Auth0）
  - [ ] SSO / SAML / OAuth
- 外部身份如何映射到内部 user_id？
  - 例：`payload.openid → system_users.openid → user_id`

### 3.2 组织结构
- 部门归属字段：`users.dept_id`
- 上下级关系字段：`depts.leader_user_id` 或 `users.manager_id`
- 多公司 / 多租户：`tenant_id` 或 `company_id`

### 3.3 主管关系
- "主管能看下属数据"如何实现？
  - [ ] 部门树 + 下属递归查询
  - [ ] 显式的 manager_id 链
  - [ ] 角色数据范围配置

## 四、KPI / 统计口径（如有考核 / 报表）

任何 KPI、财务、统计指标必须明确：

| 指标名 | 时间范围 | 人员范围 | 部门范围 | 数据来源 | 排除规则 | 含子部门 |
|---|---|---|---|---|---|---|
| 例：月度销售额 | 自然月 | 在职销售 | 销售部 | 订单表（已支付） | 退款 / 作废订单 | 是 |
|  |  |  |  |  |  |  |

**口径原则**：
- 必须可解释、可追溯
- 机器人 / 外部数据**不得**直接决定 KPI；必须经过校验、幂等、必要时人工确认
- 涉及历史补录或重算必须说明：幂等键、覆盖规则、审计日志、回滚方式

---

## 五、状态机（核心实体的状态流转）

每个核心实体（订单 / 工单 / 审批 / 任务）画状态机：

```
订单状态：
  pending → paid → shipping → delivered → completed
                ↘ canceled
  paid → refunded（部分 / 全部）
```

每个状态记录：
- 谁能触发此状态变更（角色 + 权限码）
- 触发条件
- 副作用（通知 / 库存扣减 / 积分变更）
- 是否可逆

---

## 六、数据写入规则

- 写入必须**幂等**：避免重复消息 / 重复导入 / 重复计算
- 敏感字段修改必须**可追溯**（审计日志）
- **不允许硬编码**：用户 / 员工 / 部门 / 角色 / 公司 / 租户 ID
- **不允许绕过**：租户、角色、数据权限
- **不允许删除现网数据**"修复"业务问题；只能用可回滚迁移 / 状态变更 / 补偿脚本

---

## 七、菜单与权限映射

每个管理端功能入口必须对应**菜单权限**；每个按钮动作必须对应**权限码**。

| 菜单 | 子菜单 | 按钮 | 权限码 | 角色 |
|---|---|---|---|---|
| 订单管理 | 订单列表 | 查看 | `order:view` | admin / manager / staff |
|  |  | 创建 | `order:create` | admin / manager |
|  |  | 删除 | `order:delete` | admin |
|  |  | 导出 | `order:export` | admin |

后端必须做权限判断，**前端只负责体验层控制**。

---

## 八、业务约束 / 不变量

写清业务的"铁律"，AI 改代码时不能违反：

- 例：一个用户同一商品同一时间只能有一个未支付订单
- 例：库存扣减必须事务内完成，不出现负库存
- 例：金额计算必须用 decimal，不用 float
- 例：审批流的发起人不能是自己审批

---

## 九、外部数据接入（如有）

如果有外部系统（机器人 / Webhook / 第三方回调）喂数据：

### 9.1 边界
- 外部系统**只**收集和推送数据
- 外部系统**不**定义"人是谁、属于哪个部门、KPI 多少"
- 外部数据**不直接**进核心业务表

### 9.2 标准闭环
```
外部系统 → 接口接收层 → 来源鉴权 → 格式校验 → 标准化 DTO
       → 幂等处理 → 业务中间表 / 待确认表
       → 人工确认 / 权限审批 → 核心业务表
       → 状态回写 / 看板统计 / 预警通知
```

### 9.3 必须具备
- 接口鉴权（来源识别）
- 幂等键（重复消息处理）
- 数据标准化（DTO）
- 失败日志 + 重试机制
- 写入前校验
- 写入后业务状态流转
- 数据权限归属
- 审计日志
