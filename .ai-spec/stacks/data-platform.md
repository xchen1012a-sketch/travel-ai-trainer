# 数据平台编码规范

> 适用任何数据平台项目（ETL / 数仓 / 调度 / 数据服务）。
> 核心问题：数据质量、幂等、时效性、资源消耗。

---

## 一、管道规范

### 1.1 幂等
- 每条管道（ETL / 同步 / 聚合）必须可重跑不产生重复数据
- 用分区覆盖、UPSERT、幂等键或 checkpoint 保证
- 增量管道必须记录水位（watermark），支持断点续跑

### 1.2 分批
- 大数据量表用分批处理，单批 ≤ 10000 行
- 每批打进度日志（已处理 / 总数 / 耗时 / 速度）
- 长时间管道支持暂停/恢复

### 1.3 脏数据
- 格式错误、缺失必填字段的数据不进主表
- 进 `{table}_rejected` 表，保留原始记录 + 错误原因 + 接收时间
- 脏数据率超过阈值时告警，不静默丢弃

---

## 二、数据质量

### 2.1 质量检查（管道末尾必跑）
- 行数检查（与上游对比，偏差是否在预期范围）
- 空值检查（关键字段不为 NULL）
- 主键唯一性检查
- 业务规则检查（金额非负、日期不穿越）

### 2.2 质量失败处理
- 默认阻断下游（不让脏数据流入报表/API）
- 配置质量阈值：忽略（<1%）/ 告警 / 阻断
- 质量结果写入 `data_quality_log` 表，可回溯

---

## 三、调度与时效

### 3.1 调度
- 声明每个管道的触发方式（定时 / 事件 / 手动）和执行频率
- 声明上游依赖（A 完成才能跑 B）
- 超时时间必须设置（默认 ≤ 4h），超时告警

### 3.2 时效性 SLA
- 声明每个管道的 SLO（如"T+1 08:00 前出数"）
- 延迟超过 30min 告警，超过 2h 升级

---

## 四、资源管理

- 大查询必须评估扫描量和内存消耗，加约束保护
- 禁止 `SELECT *` 无分区的全表查询
- 临时表/中间表用完即清，不留垃圾
- 生产数据源只在受控任务中使用，不暴露所有用户

---

## 五、数据服务（如有 API）

- 响应时间 P95 < 500ms
- 返回数据量有上限（默认 ≤ 1000 行）
- 查询参数必须校验（日期范围、分页、权限范围）
- 大查询走异步 + 结果缓存

---

## 六、Data Platform Review 清单

- [ ] 每条管道幂等可重跑
- [ ] 增量管道有水位和断点续跑
- [ ] 脏数据有 rejected 表记录
- [ ] 质量检查在管道末尾执行
- [ ] 质量失败策略明确（阻断/告警/忽略）
- [ ] 管道超时时间已设置
- [ ] 管道间依赖已声明
- [ ] 关键管道有 SLO 和延迟告警
- [ ] 无 `SELECT *` 全表扫
