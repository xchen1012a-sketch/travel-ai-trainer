# CLI / 命令行工具编码规范

> 适用任何 CLI 工具（Go / Rust / Python / Node.js / 其它语言）。
> 核心问题：输入不可信、输出要可管道化、错误码要准确。

---

## 一、参数与输入

### 1.1 参数设计
- 用标准参数解析库（Go flag / Rust clap / Python argparse.click / Node.js commander）
- 必选参数用位置参数，可选参数用命名 flag
- `--help` 和 `--version` 必须有
- 参数命名用 kebab-case（`--output-dir`，不用 `--outputDir`）

### 1.2 输入验证
- 所有用户输入在边界处校验（路径、端口、数字范围、格式）
- 失败立即返回非零退出码 + 具体错误信息（不泄露系统路径）
- 管道输入检测：`stdin` 是否可读、是否为空

---

## 二、输出规范

### 2.1 输出到 stdout vs stderr
- 正常结果 → stdout
- 日志/提示/错误 → stderr
- 支持 `--json` 输出结构化结果（方便管道给 jq / 下一个程序）
- 支持 `--quiet` / `--silent` 减少输出

### 2.2 管道友好
- 默认不输出进度条、颜色、交互提示（除非 stdout 是终端）
- 用 `isatty` 检测，管道模式自动降级
- 不在管道输出中混入日志

---

## 三、退出码

| 退出码 | 含义 |
|--------|------|
| 0 | 成功 |
| 1 | 一般错误 |
| 2 | 参数/输入错误 |
| 3 | 网络/外部服务错误 |
| 4 | 文件/权限错误 |

退出码必须准确，不用 0 掩盖失败。

---

## 四、配置

- 支持配置文件（yaml / json / toml）+ 环境变量 + 命令行参数
- 优先级：命令行 > 环境变量 > 配置文件 > 默认值
- 配置文件路径：默认 `./.config.yaml`，可通过 `--config` 覆盖
- 不硬编码路径、端口、域名

---

## 五、交互

- 默认非交互模式（CI / 管道友好）
- `--interactive` / `--force` 显式启用交互覆盖
- 破坏性操作（删除、清空、覆盖）必须加 `--force` 或二次确认

---

## 六、测试

- 单元测试覆盖核心逻辑
- 集成测试用 `exec` 或 subprocess 跑真实 CLI
- 测试验证退出码 + stdout + stderr
- 多平台测试（Windows 路径/换行符差异）

---

## 七、CLI Review 清单

- [ ] `--help` 可读、示例正确
- [ ] 退出码准确
- [ ] stdin/stdout/stderr 正确分流
- [ ] 管道模式不自毁输出
- [ ] 配置文件路径可覆盖
- [ ] 破坏性操作有确认
- [ ] 跨平台可运行
