# 跨端联调与交接规范

> 适用：前后端联调、服务间联调、第三方对接。
> 默认模式：**契约先行 + 分仓小步**。

---

## 一、默认模式（必走）

```
1. 先定接口契约（见 api-contract-standard.md）
2. 后端按契约实现并自测（curl / Postman / Swagger）
3. 前端按契约对**真实后端**联调（不 mock 后端）
4. 契约变更必须双向同步
```

**禁止**：
- 前端用 mock 跑通就声称完成（mock ≠ 真实后端）
- 后端没自测就推给前端"自己联调"
- 契约变更不通知对方

---

## 二、仓库模式（两种）

### 2.1 单仓模式（Monorepo）

```
my-app/
├── backend/
├── frontend/
└── shared/         # 共享类型 / 契约 / proto
```

- 同一分支同时改后端 + 前端
- 一个 MR 同时合并
- 契约放 `shared/contracts/`
- 适合：小团队、强耦合、CI 简单

### 2.2 多仓模式（Polyrepo）

```
backend-repo/   ← 后端独立 git
frontend-repo/  ← 前端独立 git
```

- 分支命名和关联方式遵循各仓配置；建议用同一需求 ID 建立可追踪关系
- 各自 MR，**互相引用**
- 契约放共享文档仓 / Wiki / `docs/contracts/`（双拷贝）
- 合并/发布顺序由兼容策略决定；优先先发布向后兼容的提供方，再切换消费方
- 适合：大团队、独立部署、技术栈差异大

---

## 三、父目录协调层（仅多仓模式）

多仓项目常有一个**父目录**作为工作区（不是 git 仓库），仅做协调：

```
my-workspace/         ← 非 git 仓库，工作区协调层
├── backend-repo/     ← 独立 git
└── frontend-repo/    ← 独立 git
```

### 父目录铁律
- **禁止**在父目录执行 `git add` / `git commit` / `git push`
- 所有 Git 写操作必须**进入对应子仓**执行
- 父目录只做：读代码、跨仓 grep、起服务、跑脚本、写文档

## 五、前端交付给后端

前端联调完成后，反馈给后端：

| 项 | 内容 |
|---|---|
| 字段使用情况 | 哪些字段实际用到、哪些没用 |
| 字段需求 | 缺哪些字段、哪些字段希望合并 |
| 性能反馈 | 接口耗时、并发调用情况 |
| Bug 清单 | 联调中发现的接口 bug |
| 体验建议 | 错误信息是否友好、状态码是否合理 |

## 六、分支与 MR（多仓模式）

### 6.1 分支命名
- 遵循各仓 `ai-spec.yaml` 和团队分支模型
- 使用共同需求 ID、契约版本或 MR/PR 互链保证跨仓追踪

### 6.2 MR 流程
```
1. 后端功能分支 → MR/PR → 合入项目配置的目标分支
2. 前端功能分支 → MR/PR → 合入项目配置的目标分支
3. 两个 MR 描述**互相引用**（"依赖 backend MR !123"）
4. 合并顺序按向后兼容和部署依赖决定，并在契约中记录
5. 不直推项目标记为受保护的分支
```

### 6.3 紧急情况
- 后端没合、前端急着测：前端用临时 mock，但 MR 描述必须标注"等后端合并后切真实"
- 上线后发现 bug：开 `hotfix/*` 分支走加速通道

## 七、AI 切换 / 任务交接

切换 AI 工具 / 会话 / 人时，必须生成 Handoff（见 `../core/ai-workflow.md` 的 Handoff 模板）。

Handoff 至少包含：
- 当前任务目标
- 当前分支
- 已完成 / 未完成
- 修改文件
- 接口契约
- 权限码
- 验证结果
- 未验证风险
- 下一步指令

---

## 八、联调检查清单

联调启动前：

- [ ] 契约文档已对齐（前后端都看过、都认可）
- [ ] 后端本地启动 + 自测通过
- [ ] 前端能连真实后端（不 mock）
- [ ] 测试账号已创建、权限已配置
- [ ] 数据前置条件已准备
- [ ] 联调环境（IP / 域名 / 证书）已就绪

联调完成前：

- [ ] 所有接口走通（不含 mock）
- [ ] 错误场景测试过（参数错 / 权限缺失 / 边界）
- [ ] 字段使用清单已同步
- [ ] Bug 已提交并跟踪
- [ ] MR 已提、分支已对齐
