# API 接口契约规范

> 适用 REST / gRPC / GraphQL / 消息队列 topic。
> 跨端协作功能动手前**必须**先定契约，再写代码。

---

## 一、契约先行（必走）

跨端（前后端 / 服务间 / 第三方对接）功能动手前，必须先定接口契约。

**契约必须包含**（11 项）：

1. 模块 / 功能名
2. URL 路径（含版本号）
3. HTTP 方法（REST）或 RPC 方法（gRPC）或 operation name（GraphQL）
4. **权限码**（鉴权要求）
5. 请求参数字段：名 / 类型 / 是否必填 / 默认值 / 取值范围
6. 响应字段：名 / 类型 / 枚举含义 / 空值语义
7. 错误码与触发条件
8. **数据权限范围**（谁能查 / 能查哪些范围）
9. 幂等性策略（高价值、可重试或外部写操作必须说明幂等键/唯一约束；其它写操作说明重复请求语义）
10. 限流 / 缓存策略
11. 示例请求 + 示例响应

---

## 二、契约文档落点

| 文档类型 | 路径 | 用途 |
|---|---|---|
| 跨前后端协作契约 | `docs/contracts/{feature}.md` | 双方对齐 |
| 单次任务交接 | `governance/handoff-template.md` 派生 | AI 切换时 |
| OpenAPI / proto | `docs/openapi.yaml` / `proto/{module}.proto` | 自动生成 |
| 运行时校验 | `/v3/api-docs`（Swagger）或 GraphQL introspection | 不能替代契约说明 |

---

## 三、REST 约定

### 3.1 路径
- 基路径由项目 `ai-spec.yaml` 或现有契约定义；常见形式为 `/{base}/{version}/{module}/{resource}`，例：`/api/v1/orders`
- 资源用**复数**：`/orders` 不是 `/order`
- 层级不超过 2 级：`/users/{id}/orders` ✅，`/users/{id}/orders/{oid}/items/{iid}/...` ❌

### 3.2 方法语义
| 方法 | 语义 | 幂等 |
|---|---|---|
| GET | 查询 | ✅ |
| POST | 创建 / 触发动作 | ❌（除非带幂等键） |
| PUT | 整体替换 | ✅ |
| PATCH | 部分更新 | 取决于操作语义，必须在契约中说明 |
| DELETE | 删除 | ✅ |

### 3.3 状态码
| 码 | 用途 |
|---|---|
| 200 | 成功（GET / PATCH） |
| 201 | 创建成功（POST） |
| 204 | 成功无内容（DELETE） |
| 400 | 参数错误 |
| 401 | 未认证 |
| 403 | 无权限 |
| 404 | 资源不存在 |
| 409 | 冲突（重复 / 状态不允许） |
| 422 | 业务校验失败 |
| 429 | 限流 |
| 500 | 服务端错误 |
| 503 | 服务不可用 |

## 四、响应包装（PROJECT 决策）

响应是否使用统一信封必须沿用项目现有契约。标准 HTTP API、文件流、GraphQL、gRPC 和事件消息不应被机械包装。采用业务信封时，推荐字段：`code`（状态码，0 成功 非 0 错误）、`data`（负载）、`msg`（描述）、分页增 `total`/`page`/`limit`。完整示例见 `contracts/examples/api-response-envelopes.md`。

---

## 五、字段规则

### 5.1 命名
- **后端字段**：按语言习惯（Java `camelCase` / Python `snake_case`）
- **前端实际 key**：明确写在契约里
- 不一致时用注解映射：Java `@JsonProperty("xxx")` / Jackson / Gson 自定义命名

### 5.2 类型与单位
- 金额：`integer`（分）或 `decimal`，**带精度说明**，禁止 float
- 比率：`integer`（万分比 / 百分比），注明
- 时间：ISO 8601 字符串 `2026-06-19T10:30:00Z` 或时间戳（注明秒 / 毫秒、时区）
- 时间范围：注明**含 / 不含**边界
- 布尔：用 `true`/`false`，不用 `0`/`1` 或 `"Y"`/`"N"`
- 枚举：用字符串字面量，不用魔法数字
- ID：用字符串（雪花 ID / UUID）或 number（自增），注明

### 5.3 空值语义
- `null`：字段存在但无值
- 缺字段：字段不存在
- 空数组 `[]` vs `null`：含义不同，注明

---

## 六、错误码规范

### 6.1 编码
- 5 位数字：`{module}{error_type}{seq}`
- 例：`10`（订单模块）+ `01`（参数错误）+ `001` = `1001001`
- 或按 HTTP 状态码前缀：`40001` = 400 类参数错误的第 1 个

### 6.2 错误码必须文档化
每个错误码在契约里写：
- code 数值
- 触发条件
- 用户可见的 msg 模板
- 给前端的处理建议

---

## 七、版本化

### 7.1 版本策略
- URL 版本：`/api/v1/...` `/api/v2/...`（推荐，简单直观）
- Header 版本：`Accept: application/vnd.x.v1+json`
- 查询参数：`?version=1`（不推荐，容易丢）

### 7.2 向后兼容规则
- **可兼容**变更（直接改）：
  - 新增可选请求字段
  - 新增响应字段
  - 新增端点
- **破坏性**变更（必须新版本）：
  - 删除字段
  - 改字段类型
  - 改字段语义
  - 改必填 / 可选

### 7.3 弃用流程
```
1. 标记旧版弃用（响应头：Deprecation: true；文档加 DEPRECATED）
2. 提供迁移指南
3. 通知所有调用方
4. 等待 N 个月（业务而定）
5. 下线旧版
```

---

## 八、契约变更规则

- 路径 / 方法 / 字段名 / 字段类型 / 权限码 / 错误码**任何变更**必须：
  1. 更新 `docs/contracts/`
  2. 同步通知前端 / 调用方
  3. 提供"变更前后对比"
- 兼容性变更优先**新增字段**，不删旧字段
- 删除或改语义必须说明影响面 + 迁移路径

---
